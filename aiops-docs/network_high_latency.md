# 网络延迟过高告警处理方案

## 告警名称
- **告警名**: `NetworkHighLatency`
- **告警级别**: 警告
- **触发条件**: 服务间网络延迟 P99 持续5分钟超过500ms

## 问题描述
当服务间的网络延迟持续超过阈值时，可能导致：
- 服务间调用超时
- 请求堆积
- 用户体验下降
- 可能触发雪崩效应

## 排查步骤

### 步骤1: 获取当前时间
**工具**: `get_current_time`
**目的**: 确定告警发生的时间范围，用于后续日志查询

### 步骤2: 查询系统日志
**工具**: `search_log` (CLS)
**参数要求**:
- **地域**: `ap-guangzhou` (广州)
- **日志主题**: `network-metrics`
- **时间范围**: 最近30分钟
- **查询条件**: `latency:>500 AND service_name:*`

**查询示例**:
```
地域: ap-guangzhou
日志主题: network-metrics
时间范围: [当前时间-30分钟] 到 [当前时间]
查询语句: latency > 500 AND service_name:*
```

### 步骤3: 分析网络延迟
从日志中提取：
- 延迟最高的服务对
- 延迟峰值时间
- 延迟变化趋势

### 步骤4: 查询应用日志
**工具**: `search_log` (CLS)
**参数要求**:
- **地域**: `ap-guangzhou`
- **日志主题**: `application-logs`
- **时间范围**: 与告警时间一致
- **查询条件**: `level:ERROR OR level:WARN`

### 步骤5: 查询系统资源使用情况
**工具**: `query_cpu_metrics` (Monitor) 和 `query_memory_metrics` (Monitor)
**参数要求**:
- **地域**: `ap-guangzhou`
- **时间范围**: 最近30分钟

## 常见原因分析

> **注意**: 网络延迟排查本质上需要网络诊断工具（ping、traceroute、netstat 等），
> 当前系统不直接提供这些工具。以下分析聚焦于通过日志和监控间接定位问题，
> 对于需要网络设备或链路层面的操作，标注为需要人工配合网络团队处理。

### 原因1: 跨地域/跨可用区网络延迟
**特征**:
- 特定服务之间延迟高于正常水平
- 应用日志中有大量 RPC 超时记录
- CPU和内存使用率正常但请求耗时增加

**处理方案**:
1. **通过日志确认延迟范围**:
   - 查询应用日志中的 RPC 调用耗时，确认是哪些服务间的调用变慢
   - 对比正常时段的调用耗时，量化延迟增幅

2. **检查流量调度**:
   - 确认负载均衡器是否将流量正确路由到就近实例
   - 检查是否有流量被错误地路由到了异地机房

3. **联系网络团队**（人工配合）:
   - 通知网络团队检查跨地域链路的带宽和延迟
   - 确认是否需要启用备用链路或临时扩容带宽

### 原因2: 应用层协议问题
**特征**:
- 日志中有大量连接超时或连接重置的记录
- 特定类型的请求（如大文件传输）延迟显著
- 数据库连接池中有大量等待获取连接的线程

**处理方案**:
1. **通过应用日志排查**:
   - 查询 `application-logs` 中的 `ConnectionTimeout` 或 `ReadTimeout` 记录
   - 确认超时配置是否合理，是否存在连锁超时

2. **优化连接管理**:
   - 调整服务间调用的连接池大小和超时时间
   - 为长耗时请求启用异步处理或消息队列解耦

3. **检查请求放大**:
   - 确认是否存在循环调用或 N+1 查询问题导致请求数放大

### 原因3: DNS解析异常
**特征**:
- DNS解析时间长
- 日志中有DNS解析失败记录
- 特定域名访问慢

**处理方案**:
1. **检查DNS配置**（人工配合）:
   - 确认DNS服务器配置正确
   - 检查DNS缓存设置是否合理

2. **优化DNS解析**:
   - 配置多个DNS服务器作为备用
   - 在应用层增加 DNS 解析结果缓存

### 原因4: 服务器资源瓶颈
**特征**:
- 服务器CPU或内存使用率高
- 应用响应时间长
- 网络延迟伴随其他性能指标恶化

**处理方案**:
1. **通过监控确认**:
   - 使用 `query_cpu_metrics` 和 `query_memory_metrics` 确认资源使用情况
   - 判断延迟是由网络还是服务器资源瓶颈引起

2. **扩容资源**:
   - 增加服务器实例数量
   - 升级服务器规格

3. **优化代码**:
   - 优化应用代码，减少资源消耗
   - 优化数据库查询

## 紧急处理措施

### 立即操作（5分钟内）
1. **确认影响范围**: 通过 `search_log` 查询最近5分钟的应用错误日志，确认哪些服务间调用受影响
2. **流量切换**: 如果监控显示单个实例或可用区延迟异常，通知网络团队将流量切到其他可用区
3. **降级非核心调用**: 对受延迟影响的服务，降级非关键的外部依赖调用

### 短期措施（30分钟内）
1. **分析日志**: 通过 `search_log` 和监控指标找出延迟集中在哪些服务间
2. **调整超时配置**: 临时调高受影响的 RPC 或 HTTP 客户端超时时间，避免连锁超时
3. **通知网络团队**: 提供具体的延迟数据，配合网络团队排查链路问题

### 长期优化
1. **架构优化**: 引入多线路、多地域部署，实现跨地域自动切换
2. **监控完善**: 增加服务间调用延迟的分位数监控，设置合理的告警阈值
3. **超时调优**: 定期审查各服务的超时配置和重试策略，避免级联超时

## 验证步骤
1. 确认网络延迟降到告警阈值以下（P99 < 500ms）
2. 检查应用响应时间恢复正常
3. 确认无新的错误日志产生
4. 观察30分钟确保问题不再复现

## 相关告警
- `HighErrorRate`: 错误率过高
- `ServiceTimeout`: 服务超时
- `HighLoadAverage`: 系统负载过高
