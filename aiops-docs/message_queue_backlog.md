# 消息队列积压告警处理方案

## 告警名称
- **告警名**: `MessageQueueBacklog`
- **告警级别**: 严重
- **触发条件**: 消息队列（Kafka/RocketMQ）消费延迟持续10分钟超过10000条

## 问题描述
当消息队列的消费者处理速度落后于生产者速度时，会导致：
- 消息堆积
- 消费延迟增加
- 业务延迟
- 可能导致下游服务响应变慢或失败

## 排查步骤

### 步骤1: 获取当前时间
**工具**: `get_current_time`
**目的**: 确定告警发生的时间范围，用于后续日志查询

### 步骤2: 查询消息队列日志
**工具**: `search_topic_by_service_name` (CLS)
**参数要求**:
- **服务名**: `kafka` 或 `rocketmq`
- **地域**: `ap-guangzhou`

**查询示例**:
```
服务名: kafka
地域: ap-guangzhou
```

### 步骤3: 查询具体的日志主题
**工具**: `get_topic_info_by_name` (CLS)
**参数要求**:
- **日志主题名**: 从上一步获取的日志主题名
- **地域**: `ap-guangzhou`

**查询示例**:
```
日志主题名: kafka-consumer-lag
地域: ap-guangzhou
```

### 步骤4: 检索相关日志
**工具**: `search_log` (CLS)
**参数要求**:
- **地域**: `ap-guangzhou`
- **日志主题**: 从上一步获取的日志主题
- **时间范围**: 最近1小时
- **查询条件**: `consumer_lag > 10000`

**查询示例**:
```
地域: ap-guangzhou
日志主题: kafka-consumer-lag
时间范围: [当前时间-1小时] 到 [当前时间]
查询语句: consumer_lag > 10000
```

### 步骤5: 分析CPU和内存使用情况
**工具**: `query_cpu_metrics` (Monitor) 和 `query_memory_metrics` (Monitor)
**参数要求**:
- **服务名**: 消费者所在的服务名
- **时间范围**: 最近1小时

**查询示例**:
```
服务名: my-consumer-service
时间范围: [当前时间-1小时] 到 [当前时间]
```

### 步骤6: 检查应用日志
**工具**: `search_topic_by_service_name` (CLS)
**参数要求**:
- **服务名**: 应用服务名
- **地域**: `ap-guangzhou`

**查询示例**:
```
服务名: my-app-service
地域: ap-guangzhou
```

### 步骤7: 检索具体的应用日志
**工具**: `search_log` (CLS)
**参数要求**:
- **地域**: `ap-guangzhou`
- **日志主题**: 从上一步获取的日志主题
- **时间范围**: 最近1小时
- **查询条件**: `level:ERROR OR level:WARN`

**查询示例**:
```
地域: ap-guangzhou
日志主题: my-app-service-logs
时间范围: [当前时间-1小时] 到 [当前时间]
查询语句: level:ERROR OR level:WARN
```

## 常见原因分析

### 原因1: 消费者处理能力不足
**特征**:
- 消费者实例数量不足
- 单个消费者处理速度慢
- 日志中显示处理超时或异常

**处理方案**:
1. **扩容消费者实例**:
   - 增加消费者实例数量
   - 调整消费者线程数
2. **优化消费者代码**:
   - 优化处理逻辑
   - 减少不必要的处理步骤
3. **监控和告警**:
   - 设置消费者处理速率监控
   - 增加处理超时告警

### 原因2: 生产者流量突增
**特征**:
- 消息生产速率突然增加
- 生产者日志中有大量发送记录
- 消费者处理速度跟不上生产速度

**处理方案**:
1. **限流生产者**:
   - 启用生产者限流机制
   - 限制单个生产者的发送速率
2. **增加消费者实例**:
   - 扩容消费者实例以应对流量突增
   - 动态调整消费者数量
3. **优化生产者代码**:
   - 避免短时间内大量发送消息
   - 使用批量发送减少消息数量

### 原因3: 消费者配置不当
**特征**:
- 消费者配置不合理
- 消费者线程数过少
- 消费者批次大小设置不合理

**处理方案**:
1. **调整消费者配置**:
   - 增加消费者线程数
   - 调整批次大小
   - 优化消费者组配置
2. **验证配置**:
   - 在测试环境验证新配置
   - 逐步上线新配置
3. **监控和告警**:
   - 设置消费者配置变更告警
   - 监控消费者处理速率

### 原因4: 系统资源不足
**特征**:
- CPU或内存使用率高
- 磁盘IO繁忙
- 网络带宽占用高
- 系统负载过高

**处理方案**:
1. **资源扩容**:
   - 增加实例数量（水平扩展）
   - 升级实例规格（垂直扩展）
   - 优化资源分配
2. **负载均衡**:
   - 检查负载均衡配置
   - 确保流量均匀分布
   - 移除异常实例
3. **限流降级**:
   - 启用限流保护
   - 降级非核心功能
   - 优先保证核心业务

### 原因5: 网络问题
**特征**:
- 跨地域调用延迟高
- 网络丢包率增加
- 特定时间段网络拥堵
- 日志中有网络超时错误

**处理方案**:
1. **网络优化**:
   - 将消费者部署在消息队列 Broker 所在可用区减少网络延迟
   - 调整消费者拉取批次大小适应网络带宽
   - 检查跨地域流量是否走了专线而非公网
2. **就近访问**:
   - 在多个地域部署消费者实例，减少跨地域网络跳数
   - 确保消费者连接的是最近的 Broker 节点
3. **压缩传输**:
   - 启用消息压缩（如 Kafka 的 compression.type=gzip）
   - 减少单条消息的体积

## 紧急处理措施

### 立即操作（5分钟内）
1. **扩容消费者实例**:
   - 如果是消费者处理能力不足，立即增加消费者实例数量
2. **限流生产者**:
   - 如果是生产者流量突增，启用生产者限流机制
3. **重启消费者实例**:
   - 如果是消费者配置不当或资源不足，重启受影响的消费者实例

### 短期措施（30分钟内）
1. **定位根因**:
   - 通过日志分析找出慢的环节
2. **临时优化**:
   - 调整消费者配置
   - 优化生产者发送逻辑
   - 增加缓存
3. **监控观察**:
   - 持续观察消息队列的消费延迟变化

### 长期优化
1. **性能优化**:
   - 代码层面的性能优化
   - 优化消费者处理逻辑
   - 减少不必要的处理步骤
2. **架构优化**:
   - 引入缓存、消息队列等
   - 优化系统资源分配
3. **监控完善**:
   - 增加更细粒度的性能监控
   - 设置消费者处理速率监控
4. **压力测试**:
   - 定期进行性能压测
   - 模拟高流量场景进行测试

## 验证步骤
1. 确认消息队列的消费延迟降到正常水平（<1000条）
2. 检查消费者处理速率是否恢复正常
3. 观察应用日志无新的错误日志产生
4. 持续监控30分钟确保稳定

## 相关告警
- `HighCPUUsage`: CPU使用率过高
- `HighMemoryUsage`: 内存使用率过高
- `SlowResponse`: 响应时间过长
- `DatabaseConnectionFailed`: 数据库连接失败
