# API 错误率飙升告警处理方案

## 告警名称
- **告警名**: `APIErrorRateSpike`
- **告警级别**: 紧急
- **触发条件**: API 5xx 错误率持续3分钟超过5%

## 问题描述
当接口返回大量 5xx 错误时，可能导致：
- 用户请求失败
- 业务中断
- 用户体验下降
- 可能引发雪崩效应

## 排查步骤

### 步骤1: 获取当前时间
**工具**: `get_current_time`
**目的**: 确定告警发生的时间范围，用于后续日志查询

### 步骤2: 搜索服务对应的日志主题
**工具**: `search_topic_by_service_name` (CLS)
**参数要求**:
- **服务名**: `api-gateway`
- **地域**: `ap-guangzhou` (广州)

**查询示例**:
```
服务名: api-gateway
地域: ap-guangzhou
```

### 步骤3: 检查日志主题信息
**工具**: `get_topic_info_by_name`
**参数要求**:
- **日志主题名**: 上一步查询到的日志主题名
- **地域代码**: `get_region_code_by_name("ap-guangzhou")`

**查询示例**:
```
日志主题名: api-gateway-logs
地域代码: 100009
```

### 步骤4: 按时间范围和查询条件检索日志
**工具**: `search_log`
**参数要求**:
- **地域**: `ap-guangzhou`
- **日志主题**: `api-gateway-logs`
- **时间范围**: 最近30分钟
- **查询条件**: `level:ERROR OR status:5xx`

**查询示例**:
```
地域: ap-guangzhou
日志主题: api-gateway-logs
时间范围: [当前时间-30分钟] 到 [当前时间]
查询语句: level:ERROR AND status:5xx
```

### 步骤5: 分析错误日志
从日志中提取：
- 错误类型
- 错误频率
- 请求路径
- 请求参数
- 错误堆栈

### 步骤6: 查询 CPU 和内存监控指标
**工具**: `query_cpu_metrics`, `query_memory_metrics`
**参数要求**:
- **地域**: `ap-guangzhou`
- **时间范围**: 最近30分钟

**查询示例**:
```
地域: ap-guangzhou
时间范围: [当前时间-30分钟] 到 [当前时间]
查询语句: cpu_usage > 80
```

## 常见原因分析

### 原因1: 上游依赖故障
**特征**:
- 日志中有上游服务调用失败记录
- 特定请求路径错误率高
- 上游服务响应超时或连接失败

**处理方案**:
1. **检查上游服务状态**:
   - 查看上游服务的健康检查
   - 检查网络连通性
   - 验证配置文件

2. **启用降级策略**:
   - 启用熔断机制
   - 返回默认值或缓存数据
   - 异步处理非关键调用

3. **恢复上游服务**:
   - 重启上游服务
   - 修复配置错误
   - 优化网络连接

### 原因2: 代码缺陷
**特征**:
- 特定代码路径频繁抛出异常
- 错误堆栈指向特定方法
- 无明显外部依赖问题

**处理方案**:
1. **定位代码问题**:
   - 使用APM工具定位慢方法
   - 查看火焰图找出热点
   - 分析代码执行路径

2. **代码优化**:
   - 修复代码bug
   - 优化循环和递归
   - 减少不必要的对象创建

3. **回滚发布**:
   - 如果是新版本引入的问题，立即回滚到上一个稳定版本
   - 使用蓝绿部署或金丝雀发布策略

### 原因3: 配置错误
**特征**:
- 最近有配置变更
- 日志中有配置加载错误
- 特定环境变量缺失
- 启动参数错误

**处理方案**:
1. **回滚配置**:
   - 恢复到上一个正确的配置
   - 检查配置文件语法
   - 验证环境变量

2. **修复配置**:
   - 对比正确的配置文件
   - 修正错误的配置项
   - 重新加载配置

3. **配置管理**:
   - 使用配置中心统一管理
   - 配置变更需要审批
   - 保留配置历史版本

### 原因4: 流量峰值
**特征**:
- 请求量突然激增
- 多个进程CPU使用率均匀升高
- 响应时间变长但无明显错误

**处理方案**:
1. **检查流量来源**:
   - 检查是否有营销活动或突发流量
   - 查看访问日志分析流量趋势

2. **启动自动扩容机制**:
   - 增加实例数量
   - 调整负载均衡器配置

3. **启用限流保护**:
   - 设置合理的限流规则
   - 优先保证核心业务

### 原因5: 网络问题
**特征**:
- 无法访问服务
- 网络连接超时
- 负载均衡器异常
- DNS解析失败

**处理方案**:
1. **检查网络**:
   - 查看应用日志中是否有大量连接超时或拒绝连接的错误
   - 检查负载均衡器健康检查状态
   - 确认上游依赖服务的健康检查接口是否正常

2. **修复网络**:
   - 更新负载均衡器配置，摘除异常后端实例
   - 检查 DNS 解析是否正常
   - 通知网络团队排查链路故障

3. **流量切换**:
   - 切换到备用可用区或地域
   - 使用备用域名指向健康集群

## 紧急处理措施

### 立即操作（5分钟内）
1. **扩容**: 如果是流量问题，立即增加实例数量
2. **限流**: 如果怀疑恶意流量，启用限流规则
3. **重启**: 如果是单个实例问题，重启该实例

### 短期措施（30分钟内）
1. **分析日志找出根本原因**:
   - 通过日志分析找出慢的环节
   - 临时优化：增加缓存、优化慢SQL、调整超时时间
2. **如果代码问题，准备回滚**:
   - 回滚到上一个稳定版本
   - 使用蓝绿部署或金丝雀发布策略
3. **如果配置问题，调整配置**:
   - 修正错误的配置项
   - 重新加载配置
4. **持续监控CPU使用率变化**:
   - 监控扩容后的CPU使用率
   - 观察响应时间和错误率

### 长期优化
1. **代码性能优化**:
   - 优化循环和递归
   - 减少不必要的对象创建
   - 使用更高效的数据结构
2. **增加监控告警阈值**:
   - 设置合理的告警阈值
   - 增加细粒度的性能监控
3. **完善自动扩缩容策略**:
   - 根据业务增长合理规划容量
   - 实现弹性扩容
4. **定期进行压力测试**:
   - 定期进行性能压测
   - 优化系统瓶颈

## 验证步骤
1. 确认API 5xx错误率降到正常水平（<1%）
2. 检查应用响应时间恢复正常
3. 确认无新的错误日志产生
4. 观察30分钟确保问题不再复现

## 相关告警
- `HighCPUUsage`: CPU使用率过高
- `HighMemoryUsage`: 内存使用率过高
- `DatabaseConnectionFailed`: 数据库连接失败
- `ServiceTimeout`: 服务超时
