# 数据库连接池耗尽告警处理方案

## 告警名称
- **告警名**: `DatabaseConnectionPoolExhaustion`
- **告警级别**: 严重
- **触发条件**: 数据库连接池活跃连接数持续5分钟超过90%

## 问题描述
当数据库连接池（如 HikariCP、Druid）中的连接耗尽时，会导致：
- 新请求无法获取数据库连接
- 请求超时增加
- 服务响应变慢
- 可能触发雪崩效应

## 排查步骤

### 步骤1: 获取当前时间
**工具**: `get_current_time`
**目的**: 确定告警发生的时间范围，用于后续日志查询

### 步骤2: 查询系统日志
**工具**: `search_topic_by_service_name`
**参数要求**:
- **服务名**: `your-app-service`（替换为实际应用服务名）

**查询示例**:
```
服务名: database-connection-pool
```

### 步骤3: 检查数据库连接池日志
**工具**: `search_log`
**参数要求**:
- **地域**: `ap-guangzhou`
- **日志主题**: `database-connection-pool`
- **时间范围**: 最近30分钟
- **查询条件**: `level:ERROR OR level:WARN`

**查询示例**:
```
地域: ap-guangzhou
日志主题: database-connection-pool
时间范围: [当前时间-30分钟] 到 [当前时间]
查询语句: (level:ERROR OR level:WARN) AND service_name:*
```

### 步骤4: 分析数据库连接池状态
从日志中提取：
- 连接池的最大连接数
- 当前活跃连接数
- 空闲连接数
- 连接等待队列长度

### 步骤5: 查询应用日志
**工具**: `search_log`
**参数要求**:
- **地域**: `ap-guangzhou`
- **日志主题**: `application-logs`
- **时间范围**: 与告警时间一致
- **查询条件**: `database_connection_error OR connection_timeout`

### 步骤6: 分析应用请求日志
从日志中提取：
- 请求量变化
- 请求超时记录
- 错误堆栈信息

### 步骤7: 查询数据库性能指标
**工具**: `query_cpu_metrics` 和 `query_memory_metrics`
**参数要求**:
- **地域**: `ap-guangzhou`
- **时间范围**: 最近30分钟

**查询示例**:
```
地域: ap-guangzhou
时间范围: [当前时间-30分钟] 到 [当前时间]
查询指标: CPU使用率, 内存使用率
```

## 常见原因分析

### 原因1: 连接泄漏
**特征**:
- 连接池中的空闲连接数持续减少
- 应用日志中有大量连接超时记录
- 数据库连接数接近最大值

**处理方案**:
1. **检查代码**: 查找未关闭的数据库连接
2. **优化代码**: 确保所有连接在使用后被正确关闭
3. **重启应用**: 释放未关闭的连接
4. **监控连接数**: 设置连接数监控和告警

### 原因2: 流量突增
**特征**:
- 连接池活跃连接数突然增加
- 请求量激增
- 应用响应时间变长

**处理方案**:
1. **扩容实例**: 增加应用实例数量
2. **启用限流**: 保护系统不被压垮
3. **优化缓存**: 减少对数据库的直接访问
4. **调整连接池配置**: 增加最大连接数

### 原因3: 数据库性能问题
**特征**:
- 数据库CPU或内存使用率高
- 慢查询日志中有大量记录
- 数据库连接池接近满载

**处理方案**:
1. **优化SQL**: 找出并优化慢查询
2. **添加索引**: 对常用查询字段添加索引
3. **分库分表**: 如果是数据量过大，考虑分库分表
4. **增加数据库资源**: 升级数据库实例规格

### 原因4: 配置不当
**特征**:
- 连接池配置不合理
- 最大连接数设置过低
- 连接超时时间设置过短

**处理方案**:
1. **调整连接池配置**:
   - 增加最大连接数
   - 调整连接超时时间
   - 优化连接池其他参数
2. **验证配置**: 通过压力测试验证配置合理性
3. **文档更新**: 更新配置文档，确保配置合理

### 原因5: 网络问题
**特征**:
- 网络延迟高
- 数据库连接不稳定
- 日志中有网络超时错误

**处理方案**:
1. **检查网络**: 
   - 通过应用日志确认是否存在数据库连接超时
   - 检查应用与数据库是否在同一可用区，跨可用区会增加 1-3ms 延迟
   - 确认安全组和防火墙未限制数据库端口
2. **优化网络**: 
   - 将数据库与应用部署在同一可用区减少网络跳数
   - 使用数据库读写分离，将读请求路由到就近只读实例
   - 调整连接池的 TCP keepalive 参数缩短僵死连接检测时间

## 紧急处理措施

### 立即操作（5分钟内）
1. **重启应用**: 释放未关闭的连接
2. **扩容实例**: 增加应用实例数量
3. **启用限流**: 保护系统不被压垮

### 短期措施（30分钟内）
1. **分析日志**: 确定连接池耗尽的根本原因
2. **优化配置**: 调整连接池配置
3. **优化代码**: 修复连接泄漏问题
4. **监控观察**: 持续监控连接池状态

### 长期优化
1. **代码优化**: 修复连接泄漏问题
2. **架构优化**: 引入缓存、消息队列等
3. **监控完善**: 增加更细粒度的连接池监控
4. **压力测试**: 定期进行性能压测

## 验证步骤
1. 确认数据库连接池活跃连接数降到正常水平
2. 检查应用响应时间恢复正常
3. 确认无新的连接超时日志产生
4. 观察30分钟确保问题不再复现

## 相关告警
- `HighCPUUsage`: CPU使用率过高
- `HighMemoryUsage`: 内存使用率过高
- `DatabaseSlowQuery`: 数据库慢查询
- `HighErrorRate`: 错误率过高
