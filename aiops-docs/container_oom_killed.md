# 容器 OOM 被杀告警处理方案

## 告警名称
- **告警名**: `ContainerOOMKilled`
- **告警级别**: 紧急
- **触发条件**: 容器被 OOM Killer 终止（exit code 137）

## 问题描述
当 K8s Pod 因超出内存 limit 被 OOMKilled 时，会导致：
- 容器重启
- 服务短暂不可用
- 可能影响用户体验
- 可能导致数据丢失或不一致

## 排查步骤

### 步骤1: 获取当前时间
**工具**: `get_current_time`
**目的**: 确定告警发生的时间范围，用于后续日志查询

### 步骤2: 查询系统监控日志
**工具**: `query_memory_metrics` (Monitor)
**参数要求**:
- **地域**: `ap-guangzhou` (广州)
- **时间范围**: 最近30分钟
- **查询条件**: 内存使用率超过90%

**查询示例**:
```
地域: ap-guangzhou
时间范围: [当前时间-30分钟] 到 [当前时间]
查询语句: memory_usage > 90
```

### 步骤3: 查询容器日志
**工具**: `search_topic_by_service_name` (CLS), `search_log` (CLS)
**参数要求**:
- **地域**: `ap-guangzhou`
- **服务名**: `your-app-service`（替换为实际服务名）
- **时间范围**: 与告警时间一致
- **查询条件**: `level:ERROR OR level:WARN OR event:OOM`

**查询示例**:
```
地域: ap-guangzhou
服务名: your-app-service
时间范围: [当前时间-30分钟] 到 [当前时间]
查询语句: (level:ERROR OR level:WARN) AND event:OOM
```

### 步骤4: 分析内存使用情况
从日志中提取：
- 容器内存使用情况
- OOM 发生的具体时间
- 相关的进程信息
- 日志中的错误堆栈

## 常见原因分析

### 原因1: 内存泄漏
**特征**:
- 内存使用率持续缓慢上升
- Full GC后内存无法释放
- 应用运行时间越长，内存占用越高
- 日志中有大量GC记录

**处理方案**:
1. **紧急措施**: 重启受影响的容器实例
2. **获取堆转储**: 在重启前dump内存快照
3. **分析内存泄漏**: 通过日志中的 GC 记录和错误堆栈定位可疑的对象创建点
4. **定位代码问题**: 找出持有大量对象引用的代码
5. **修复并发布**: 修复内存泄漏代码并重新部署

### 原因2: 流量突增导致对象激增
**特征**:
- 内存使用率突然升高
- 与流量增长同步
- GC能够回收大部分内存
- 应用日志显示请求量激增

**处理方案**:
1. **立即扩容**: 增加应用实例数量
2. **启用限流**: 保护系统不被压垮
3. **优化缓存**: 检查缓存策略是否合理
4. **增加内存**: 如果是合理的业务增长，考虑增加内存配置

### 原因3: 缓存配置不当
**特征**:
- 缓存占用内存过大
- 缓存命中率低
- 缓存数据长时间不过期
- 日志中有缓存相关警告

**处理方案**:
1. **调整缓存大小**: 减少缓存容量上限
2. **优化过期策略**: 设置合理的TTL
3. **启用LRU淘汰**: 确保缓存能自动淘汰旧数据
4. **监控缓存指标**: 持续观察缓存命中率和内存占用

### 原因4: 大文件或大数据处理
**特征**:
- 特定时间点内存突增
- 日志中有文件上传或数据导入记录
- 处理完成后内存可以释放
- 与定时任务或用户操作相关

**处理方案**:
1. **优化处理逻辑**: 改为流式处理，避免一次性加载
2. **限制文件大小**: 设置上传文件大小限制
3. **分批处理**: 大数据处理改为分批次进行
4. **异步处理**: 将大数据处理任务放到后台队列

### 原因5: 运行时内存参数配置不合理
**特征**:
- 应用最大内存限制配置过小
- 垃圾回收器选择不当
- 堆外内存或直接内存溢出

**处理方案**:
1. **调整内存限制**: 根据应用实际内存使用情况合理设置容器 memory limit（建议为正常用量的 1.5-2 倍）
2. **优化运行时参数**: 对于 Java 应用调整 -Xmx/-Xms，对于 Node.js 调整 --max-old-space-size，对于 Go 调整 GOMEMLIMIT
3. **增加元空间**: 如使用 JVM 且出现 Metaspace OOM，调整 -XX:MetaspaceSize 参数

## 紧急处理措施

### 立即操作（5分钟内）
1. **重启高内存实例**: 快速释放内存，恢复服务
2. **扩容**: 增加实例数量，分散内存压力
3. **限流**: 如果是流量问题，启用限流保护

### 短期措施（30分钟内）
1. **分析日志**: 确定内存增长的根本原因
2. **dump内存快照**: 在重启前保留内存现场用于后续分析
3. **调整内存参数**: 如果是配置问题，临时调整内存限制参数
4. **清理缓存**: 手动清理不必要的缓存数据

### 长期优化
1. **代码优化**: 修复内存泄漏，优化对象创建
2. **监控完善**: 增加内存监控指标和告警
3. **压力测试**: 定期进行内存压力测试
4. **容量规划**: 根据业务增长合理规划内存容量

## 验证步骤
1. 确认内存使用率降到正常水平（<70%）
2. 观察GC频率和耗时是否正常
3. 检查应用日志无OOM错误
4. 持续监控30分钟确保稳定

## 相关告警
- `HighMemoryUsage`: 内存使用率过高
- `FrequentGC`: GC频率过高
- `OOMError`: 内存溢出错误
- `SlowResponse`: 响应时间过长
