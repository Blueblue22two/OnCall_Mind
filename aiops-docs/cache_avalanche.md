# 缓存雪崩/击穿告警处理方案

## 告警名称
- **告警名**: `CacheAvalanche`
- **告警级别**: 紧急
- **触发条件**: 缓存命中率骤降30%以上且数据库QPS飙升3倍

## 问题描述
当大量缓存 key 同时过期或热点 key 失效时，请求会直接穿透到数据库，导致：
- 数据库压力剧增
- 服务响应变慢
- 可能引发系统雪崩效应
- 用户体验下降

## 排查步骤

### 步骤1: 获取当前时间
**工具**: `get_current_time`
**目的**: 确定告警发生的时间范围，用于后续日志查询

### 步骤2: 查询缓存服务日志主题
**工具**: `search_topic_by_service_name` (CLS)
**参数要求**:
- **服务名**: `cache-service`
- **地域**: `ap-guangzhou`

**查询示例**:
```
服务名: cache-service
地域: ap-guangzhou
```

### 步骤3: 按时间检索缓存命中率日志
**工具**: `search_log` (CLS)
**参数要求**:
- **地域**: `ap-guangzhou`
- **日志主题**: 步骤2返回的日志主题名
- **时间范围**: 最近30分钟
- **查询条件**: `hit_rate:<70`

**查询示例**:
```
地域: ap-guangzhou
日志主题: database-metrics
时间范围: [当前时间-30分钟] 到 [当前时间]
查询语句: qps > 3000 AND service_name:*
```

### 步骤4: 分析缓存和数据库日志
从日志中提取：
- 缓存失效的具体key
- 数据库查询的SQL语句
- 请求量激增的时间点
- 相关的应用日志

### 步骤5: 查询应用日志
**工具**: `search_log`
**参数要求**:
- **地域**: `ap-guangzhou`
- **日志主题**: `application-logs`
- **时间范围**: 与告警时间一致
- **查询条件**: `level:ERROR OR level:WARN`

**查询示例**:
```
地域: ap-guangzhou
日志主题: application-logs
时间范围: [当前时间-30分钟] 到 [当前时间]
查询语句: (level:ERROR OR level:WARN) AND service_name:*
```

## 常见原因分析

### 原因1: 大量缓存同时过期
**特征**:
- 缓存命中率骤降
- 特定时间点缓存key大量失效
- 日志中有大量缓存miss记录
- 数据库QPS激增

**处理方案**:
1. **立即预热缓存**:
   - 重启后立即预热热点数据
   - 定时刷新缓存
   - 避免缓存同时失效

2. **优化缓存策略**:
   - 设置合理的过期时间
   - 使用随机过期时间
   - 实现缓存更新机制

3. **增加缓存容量**:
   - 如果是正常业务增长，考虑扩容缓存
   - 优化缓存使用逻辑

### 原因2: 热点key失效
**特征**:
- 某个热点key突然失效
- 数据库特定查询激增
- 日志中有大量该key的miss记录
- 应用响应时间变长

**处理方案**:
1. **临时增加缓存**:
   - 临时增加该key的缓存时间
   - 手动刷新缓存

2. **优化热点key管理**:
   - 对热点key设置更长的过期时间
   - 使用分布式锁防止并发更新
   - 实现本地缓存作为后备

3. **增加缓存层**:
   - 考虑使用多级缓存
   - 使用读写分离策略

### 原因3: 缓存配置不当
**特征**:
- 缓存容量不足
- 缓存过期时间不合理
- 缓存命中率持续低
- 日志中有频繁的缓存miss记录

**处理方案**:
1. **调整缓存大小**:
   - 增加缓存容量
   - 优化缓存存储策略

2. **优化过期策略**:
   - 设置合理的TTL
   - 使用LRU淘汰策略
   - 避免缓存集中过期

3. **监控缓存指标**:
   - 持续观察缓存命中率
   - 设置告警阈值
   - 定期清理无效缓存

### 原因4: 代码逻辑问题
**特征**:
- 应用日志中有异常堆栈
- 缓存key生成逻辑错误
- 数据库查询逻辑异常
- 缓存未正确更新

**处理方案**:
1. **修复代码问题**:
   - 查看应用日志定位代码问题
   - 回滚到上一个稳定版本
   - 修复并重新部署

2. **优化代码逻辑**:
   - 优化缓存key生成逻辑
   - 确保缓存更新逻辑正确
   - 避免不必要的缓存访问

3. **增加日志记录**:
   - 增加关键操作的日志记录
   - 便于后续排查问题

### 原因5: 网络故障
**特征**:
- 缓存服务器不可达
- 网络延迟高
- 日志中有网络超时错误
- 缓存命中率骤降

**处理方案**:
1. **检查网络连通性**:
   - 测试缓存服务器的网络连通性
   - 检查防火墙规则
   - 验证安全组配置

2. **恢复网络连接**:
   - 重启网络服务
   - 修正路由配置
   - 更新负载均衡器配置

3. **切换备用缓存**:
   - 切换到备用地域
   - 使用备用缓存节点
   - 启用多级缓存降级

## 紧急处理措施

### 立即操作（5分钟内）
1. **紧急回滚**:
   - 如果是新版本发布后出现，立即回滚到上一个稳定版本
2. **手动刷新缓存**:
   - 重启缓存服务，手动刷新热点数据
3. **启用限流**:
   - 如果怀疑恶意流量，启用限流规则
4. **增加实例**:
   - 如果是资源不足，立即增加缓存实例数量

### 短期措施（30分钟内）
1. **定位根因**:
   - 通过日志分析找出缓存失效的原因
2. **临时优化**:
   - 增加缓存容量
   - 优化缓存过期时间
   - 修复代码逻辑问题
3. **监控观察**:
   - 持续观察缓存命中率和数据库QPS变化

### 长期优化
1. **性能优化**:
   - 优化缓存策略
   - 优化代码逻辑
   - 优化数据库查询
2. **架构优化**:
   - 引入多级缓存
   - 实现缓存更新机制
   - 使用读写分离策略
3. **监控完善**:
   - 增加缓存命中率和数据库QPS的监控
   - 设置合理的告警阈值
4. **定期演练**:
   - 定期进行缓存失效演练
   - 提升系统的容错能力

## 验证步骤
1. 确认缓存命中率恢复正常（>80%）
2. 检查数据库QPS是否降到正常水平
3. 观察应用响应时间是否恢复正常
4. 持续监控30分钟确保问题不再复现

## 相关告警

- `LowCacheHitRate`: 缓存命中率过低
