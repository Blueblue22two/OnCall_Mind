# 简历格式 - 紧凑版（一个亮点 + 一段文字）

## 📌 格式说明

每个技术亮点采用：
- **标题**（加粗，10-15字）
- **一段连贯文字**（80-120字），包含：技术方案 + 实现细节 + 量化结果

---

## 🎯 中文版简历亮点

### 项目名称
**基于多智能体编排与混合RAG的自主化AIOps诊断系统**

---

### 核心技术亮点

#### **1. LangGraph Agent 编排引擎**
设计并实现基于 LangGraph StateGraph 的生产级 Agent 编排引擎，构建 Planner → Executor → Replanner 三节点有向图工作流，通过条件路由实现动态分支决策。集成 Redis 检查点持久化机制实现跨实例会话共享与故障恢复，支持 1000+ 并发会话独立状态管理。通过 SSE 流式传输实时推送 Agent 推理过程（plan/step/report 阶段），单实例支持 50+ QPS 并发查询，P95 延迟 <2.5s。

#### **2. 混合检索与重排优化**
架构混合检索管道，结合稠密向量检索（DashScope text-embedding-v4, 1024维）和稀疏 BM25 检索，使用 RRF（k=60）融合算法。在 80 条人工标注测试集上（RAGAs 评估框架），Context Precision 提升 35%（0.65 → 0.88），Context Recall 提升 28%（0.70 → 0.90），Hit Rate@3 提升 42%（0.55 → 0.78）。通过 Milvus 服务端 BM25 Function 消除客户端编码开销，检索延迟降低 40%（P95: 450ms → 270ms）。

#### **3. MCP 工具编排与弹性调用**
构建 MCP（Model Context Protocol）工具编排层，设计全局单例客户端管理器支持 10+ 分布式工具动态加载。实现指数退避重试拦截器（exponential backoff, max_retries=3），工具调用成功率从 78% 提升至 94%。采用"错误即结果"模式（error-as-result）避免异常传播导致 Agent 中断，通过拦截器模式实现横切关注点（日志、监控、限流）无需修改工具代码，支持工具调用链路追踪记录服务器、参数、耗时、重试次数。

#### **4. Token 感知的上下文管理**
工程化 Token 感知的上下文窗口管理，基于 tiktoken（cl100k_base 编码）实现精确 token 计数。采用"保留系统提示 + 滑动窗口"策略，从新到旧裁剪（LIFO）优先保留最近对话，动态裁剪历史消息保持 <8K tokens，支持 20+ 轮对话不溢出。相比固定消息数裁剪（如"保留最后 10 条"），token 裁剪节省 35% 上下文空间，显著提升多轮对话质量。

#### **5. Plan-Execute 状态机内存管理**
实现 Plan-Execute-Replan 状态机的增量式内存管理，设计紧凑状态结构（input + plan + past_steps + response，<10KB/任务）。采用增量更新策略每步仅追加 (step, result) 元组避免全量状态复制，实现结果截断机制（每步保留前 300 字符）防止长输出撑爆上下文。强制步骤数上限（MAX_STEPS=8）防止无限循环，支持 100+ 并发诊断任务内存占用 <500MB，单任务平均耗时 <3 分钟。

#### **6. 自主诊断与业务价值**
实现自主诊断 Agent 能够自主进行故障调查：查询监控系统、分析日志、关联指标，生成结构化 Markdown 诊断报告（包含根因分析、处理建议）。在 15 个故障场景测试中，工具调用准确率 92%（正确选择工具并传参），目标达成率 87%（成功生成诊断报告）。平均诊断时间从人工 15 分钟降至 Agent 3 分钟，MTTR 降低 80%，7×24 小时自动化分诊覆盖 15+ 故障场景。

## 🎯 英文版简历亮点

### Project Title
**Autonomous AIOps Agent System with Multi-Agent Orchestration & Hybrid RAG**

---

### Key Technical Highlights

#### **1. LangGraph Agent Orchestration Engine**
Architected and implemented production-grade agent orchestration engine based on LangGraph StateGraph, building a 3-node directed graph workflow (Planner → Executor → Replanner) with conditional routing for dynamic branching. Integrated Redis checkpointing for cross-instance session sharing and fault recovery, supporting 1000+ concurrent sessions with independent state management. Enabled real-time agent reasoning streaming via SSE (plan/step/report stages), achieving 50+ QPS throughput with P95 latency <2.5s per single instance.

#### **2. Hybrid Retrieval & Reranking Optimization**
Architected hybrid retrieval pipeline combining dense vector search (DashScope text-embedding-v4, 1024-dim) and sparse BM25 retrieval with RRF fusion (k=60). On 80 human-annotated test cases (RAGAs evaluation framework), achieved 35% improvement in Context Precision (0.65 → 0.88), 28% in Context Recall (0.70 → 0.90), and 42% in Hit Rate@3 (0.55 → 0.78). Leveraged Milvus server-side BM25 Function to eliminate client-side encoding overhead, reducing retrieval latency by 40% (P95: 450ms → 270ms).

#### **3. MCP Tool Orchestration & Resilient Invocation**
Built MCP (Model Context Protocol) tool orchestration layer with global singleton client manager supporting dynamic loading of 10+ distributed tools. Implemented exponential backoff retry interceptor (max_retries=3), improving tool call success rate from 78% to 94%. Adopted error-as-result pattern to prevent exception propagation from interrupting agent execution. Leveraged interceptor pattern for cross-cutting concerns (logging, monitoring, rate limiting) without modifying tool code, enabling tool call tracing with server, arguments, latency, and retry count logging.

#### **4. Token-Aware Context Window Management**
Engineered token-aware context window management with precise token counting via tiktoken (cl100k_base encoding). Adopted "preserve system prompt + sliding window" strategy with LIFO trimming (newest to oldest) to prioritize recent context. Dynamically trimmed message history to maintain <8K tokens, supporting 20+ turn conversations without overflow. Achieved 35% context space savings vs. fixed message count trimming (e.g., "keep last 10 messages"), significantly improving multi-turn conversation quality.

#### **5. Plan-Execute State Machine Memory Management**
Implemented incremental memory management for Plan-Execute-Replan state machine with compact state structure (input + plan + past_steps + response, <10KB per task). Adopted incremental update strategy appending (step, result) tuples per step to avoid full state copies. Implemented result truncation (first 300 chars per step) to prevent long outputs from exhausting context. Enforced step count limit (MAX_STEPS=8) to prevent infinite loops, supporting 100+ concurrent diagnostic tasks with <500MB memory footprint and <3 min average task duration.

#### **6. Autonomous Diagnosis & Business Impact**
Implemented autonomous diagnostic agent capable of self-directed incident investigation: querying monitoring systems, analyzing logs, correlating metrics, and generating structured Markdown diagnostic reports with root cause analysis and remediation recommendations. In 15 fault scenario tests, achieved 92% tool call accuracy (correct tool selection and parameterization) and 87% goal completion rate (successful report generation). Reduced average diagnosis time from 15 min (manual) to 3 min (agent), achieving 80% MTTR reduction with 24/7 automated triage covering 15+ fault scenarios.

## 💡 格式优势

### ✅ 为什么这种格式更好？

1. **更易阅读**：一段连贯文字比多个 bullet points 更流畅
2. **信息密度高**：80-120 字可以包含完整的"技术方案 + 实现细节 + 量化结果"
3. **专业感强**：像技术博客/论文摘要，而非简单罗列
4. **节省空间**：6 个亮点控制在 1 页内
5. **突出重点**：标题加粗，快速定位关键能力

### ⚠️ 注意事项

1. **每段控制在 80-120 字**：太短信息不足，太长难以阅读
2. **包含 3 个要素**：
   - 技术方案（做了什么）
   - 实现细节（怎么做的）
   - 量化结果（效果如何）
3. **避免重复**：每段聚焦一个核心能力
4. **数字醒目**：用百分比、具体数值增强说服力

## 🎯 快速使用指南

### 步骤 1：选择 4-6 个亮点

根据目标岗位选择：
- **偏工程岗位**：编排引擎 + 工具编排 + 上下文管理 + 业务价值
- **偏研究岗位**：混合检索 + 上下文管理 + 状态机管理 + 编排引擎
- **全栈展示**：全部 6 个亮点

### 步骤 2：复制对应段落

直接复制本文档中的段落到简历

### 步骤 3：微调细节

- 替换具体数字（如果你有真实评估数据）
- 调整技术栈（如果使用了不同的工具）
- 根据简历篇幅调整段落长度
