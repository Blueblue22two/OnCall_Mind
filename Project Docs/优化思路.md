# 1.0 React范式 （RAG-Chat-agent）优化：
- 加入Human-in-the-Loop（HITL） :当前 ReAct 完全自主运行，但运维场景中某些操作（如重启服务、切换流量）需要人工确认。
改进方案：利用 create_agent 的 interrupt_before 和 interrupt_after 参数：
self.agent = create_agent(
    model=self.model,
    tools=all_tools,
    interrupt_before=["tools"],  # 每次工具调用前暂停，等待人工批准
    checkpointer=self.checkpointer,
)
结合 LangGraph 的 graph.resume() 机制可实现"确认后继续"的人机协同。
在工具调用前插入人工审批断点，Agent 暂停执行并等待操作者确认或拒绝，实现"确认后继续"的人机协同。
补充：如果加入HITL需要在SSE的基础上加入WebSocket

# 3.0 诊断功能优化：
1. 告警触发器缺失：当前系统依赖用户主动发起诊断请求。建议增加 Webhook 监听或定时巡检机制，实现告警自动触发诊断。
2. 诊断结果闭环：建议增加诊断结果→自动执行修复动作的能力（需结合审批流程）。（该功能优先级较高）


# 4.0 LLM优化：
1. 模型配置热更新：当前 temperature 等参数硬编码在各节点中，建议统一到 `config.py` 管理。
2. Fallback 模型：单一依赖 qwen-max 存在单点风险，建议增加 qwen-plus 作为降级备选。
3. Judge 解耦：当前 Judge 模型 API Base 硬编码为第三方代理，建议支持配置多个 Judge 后端。
4. LLM API 超时配置：为 `ChatQwen` / `ChatOpenAI` 增加 `timeout` 参数（如 60s）和 `max_retries`（如 2 次）。
5. LLM Fallback 链路：qwen-max → qwen-plus 自动降级，保证服务可用性。
6. Metrics 上报：增加 LLM API 调用延迟 / 成功率 / 重试次数的 Prometheus 指标并记录，便于监控和告警。



# 5.0 Redis优化：
1. 序列化优化：LangGraph 状态快照可能较大，建议增加压缩层（如 zlib）减少 Redis 内存占用。
2. 默认启用 Redis：建议在 Docker Compose 中增加 Redis 容器，并在 `.env` 中预置 `REDIS_URL=redis://localhost:6379`。


# 6.0 Memory管理：
1. 会话超时清理：MemorySaver 模式下无自动过期，建议增加定时清理逻辑（或默认使用 RedisSaver + TTL）。
2. 摘要压缩作为可选增强：对于超过 50 轮的极长对话，可以在 token 裁剪前先生成早期对话摘要。
3. 自适应策略：根据对话轮数自动切换——<20 轮用滑动窗口，20-50 轮用摘要压缩
4. 优先级裁剪细化：当前只保留 SystemMessage，可扩展为按消息类型（工具调用结果 > 用户问题 > 助手回复）设定保留优先级。
（optional function：下面列出的几个关于长期记忆的功能实现考虑为optional）
- 引入长期记忆层：将用户偏好、历史诊断经验、常见故障模式等持久化到独立的长期记忆存储中
- 向量化的长期记忆：将历史对话和诊断结果向量化后存入 Milvus，后续可以通过语义检索召回相关经验
- Reflection 机制：让 Agent 在对话结束后"反思"本次交互，提取关键信息存入长期记忆


# 7.0 RAG优化思路：
1. 引入Query路由：根据查询类型（关键词查询 / 语义查询 / 混合查询）动态选择retrieval策略
2. 多路检索扩展：除 Dense 和 BM25 外，增加 metadata 过滤（如按告警类型筛选）
3. 提升 nprobe：将 `nprobe` 从 16 提升到 32 或 64，以牺牲少量延迟换取更高召回率
4. 引入 HNSW 索引：对于小规模数据集，HNSW 索引比 IVF\_FLAT 有更高的召回率（接近精确搜索）
5. 减少精排候选数：将 `RERANK_COARSE_TOP_K` 从 20 降到 10，精排耗时减半
6. 异步并行化：Stage 1（Rewrite）和 Stage 0（Embedding）可以并行执行——Rewrite 用原始文本，Embedding 也用原始文本，改写完成后再用改写文本做第二次 Embedding
7. Cross-Encoder 模型量化：使用 INT8 或 FP16 量化减少推理耗时
8. 弥合用户表述与文档表述的语义鸿沟：运维人员通常用口语描述问题（"服务挂了"），但知识库文档使用规范术语（"服务不可用"）。Rewrite 将口语映射到规范表述，提升向量检索的匹配度。
9. 补全缺失关键词：用户可能只描述了症状（"接口变慢了"），Rewrite 可以补充相关技术术语（"RT 升高、慢 SQL、缓存击穿"），增加检索到正确文档的概率。


# 8.0 Agent优化思路：
1. 当前两个 Agent 完全独立，未来可考虑引入 Supervisor Agent 做路由——用户输入先经过分类器，自动判断走 RAG 还是 AIOps 路径。
2. AIOps Agent 的 Planner 节点已经调用了 `retrieve_knowledge` 工具来获取知识库经验，但 RAG Agent 和 AIOps Agent 之间没有直接的数据交互，可以考虑共享知识库检索结果以减少重复调用。
3. RAG Agent 可以考虑加入 Adaptive Workflow：对于简单问题直接 ReAct，对于复杂问题（如需要多工具联合查询）切换到计划模式
